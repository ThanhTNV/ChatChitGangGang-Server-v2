# Implement from next-step spec

## Relationship to other project assets

- **Spec source:** `output/YYYY-MM-DD/next-step-spec.md` (generated by the **check-implementation-progress** skill).
- **Roadmap / checkboxes:** `output/backend-golang-implementation-plan.md`.
- **Go standards:** read and follow `.cursor/rules/golang-production.mdc` for all `*.go` changes.

If `next-step-spec.md` is missing or stale, run **check-implementation-progress** first (or ask the user which dated folder to use).

## Resolve which spec to run

1. If the user names a date or path, use **`output/THAT-DATE/next-step-spec.md`**.
2. Otherwise, pick the **most recent** `output/YYYY-MM-DD/` directory (lexicographic ISO dates sort correctly).
3. Read **`README.md`** and **`progress-summary.md`** in that folder for context if present.
4. Read **`next-step-spec.md` end-to-end** before editing code.

## Execution order

1. **Prerequisites** — confirm or note gaps (running Postgres/Redis, env files). Implement only what the spec requires; do not start dependent features early.
2. **Implementation steps** — follow the numbered steps in `next-step-spec.md` in order. Stick to the **Files to add or modify** table unless the spec explicitly allows discovery.
3. **Scope** — honor **Out of scope** in the spec. No drive-by refactors outside listed paths.
4. **OpenAPI** — if the change adds or alters HTTP endpoints, request/response JSON, query params, or auth: update `internal/apiembed/openapi.yaml` in the same PR/slice (see `.cursor/rules/openapi-sync.mdc`). Skip only when the spec explicitly excludes API surface changes.
5. **Tests** — add tests the spec requires; run `go test ./...` and `go test -race ./...` on touched packages when feasible.
6. **Lint** — run `golangci-lint run` if configured; otherwise `go vet ./...`.

## After code is in place

1. **Definition of done** — verify every bullet in `next-step-spec.md` **Definition of done** (including OpenAPI updates for any new or changed HTTP API in `internal/apiembed/openapi.yaml`).
2. **Plan sync** — if the spec says to update checkboxes, set the matching items to `- [x]` in `output/backend-golang-implementation-plan.md` (only for work actually completed).
3. **Summary for the user** — short list: files changed, how to run locally, any env vars, follow-ups or blockers.

## If the spec conflicts with the repo

- Prefer **the spec** for *this slice* if it was written after the plan.
- If the conflict is architectural or unsafe, **stop** and state the conflict; suggest regenerating the spec via **check-implementation-progress** or a one-line user decision.

## Anti-patterns

- Do not skip reading `next-step-spec.md` and invent a different “next step.”
- Do not mark plan checkboxes done without matching code/tests.
- Do not add HTTP routes without updating `internal/apiembed/openapi.yaml` when the route is part of the public API.
- Do not log secrets, tokens, or presigned URL query strings.
